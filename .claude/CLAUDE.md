# Unicorn Claude Guidance

## Rules

Before writing or modifying code that uses discord.js, zod, Prisma, or other external dependency APIs, resolve the library in Context7 and query the relevant API docs. Do not rely on training data for these libraries — always verify against current documentation.

When working with sparks, guards, configuration, or their tests, read `.claude/spark-reference.md` and `.claude/testing-reference.md` first. When modifying a core module or built-in, read the corresponding `docs/*.md` file first.

## Project Overview

Unicorn is a Discord bot framework built on Discord.js v14 and TypeScript, designed to run on Bun. It uses a "Spark" system for modular command/event handling with composable Guards for validation.

**Zod v4:** Use Zod 4 APIs — e.g. `z.url()`, `z.email()`, `z.uuid()` as standalone schemas instead of deprecated `z.string().url()` / `.email()` / `.uuid()` chains.

## Bun Runtime

- Use `bun` / `bun test` / `bun install` / `bun run <script>` — never node/npm/yarn/jest/vitest
- Bun auto-loads `.env` — don't use dotenv
- Prefer `Bun.serve()` over express, `Bun.file` over `node:fs` readFile/writeFile, `Bun.Glob` for file matching

## Quality Gate

Before every commit, run each step in order and fix issues before proceeding to the next (Do NOT use `bun qa`):

```bash
bun qa:format      # 1. Autoformat (may modify files)
bun qa:lint        # 2. Lint check
bun qa:tsc         # 3. TypeScript typecheck
bun test           # 4. Run tests (100% coverage threshold)
```
Offer to run a coderabbit review allowing user to select scope.

## Branching & Commits

**Branch model:** Git Flow lite — `main` is releases only, `develop` is the integration branch. Feature branches are created off `develop` and merged back into `develop`. Only `develop` merges into `main` for releases.

Never commit directly to `main` or `develop` — always create a descriptive feature branch off `develop`. Do not push; the user handles pushes and PRs.

**Changelog:** `CHANGELOG.md` is auto-generated by git-cliff from commit messages. The `ci-changelog` workflow updates `## [Unreleased]` on every push to `develop`. The `ci-release` workflow (manual dispatch) creates a release PR that versions the changelog and bumps `package.json`. Only `new`, `improve`, and `fix` commits appear in the changelog — `chore`, `doc`, `ci` are skipped.

**Commit format:** `<emoji> <type>([scope]): <description>` — 50 char max, imperative, lowercase. Types: `new` 🦄 / `improve` 🌈 / `fix` 🦠 / `chore` 🧺 / `release` 🚀 / `doc` 📖 / `ci` 🚦. Scope from files changed; omit if broad.

## Conventions

- `import type { ... }` for type-only imports (`verbatimModuleSyntax` enabled)
- Node built-ins use `node:` prefix — Zod: `import * as z from 'zod'` — Sentry: `import * as Sentry from '@sentry/bun'`
- Biome enforces: **kebab-case** filenames, **no `console`** (use `client.logger`), **no `process.env`** (use config/`Bun.env`), **no floating promises**, single quotes, organized imports
- Use `AppError` for domain errors. Use `attempt()` for all fallible/async calls unless there is a compelling reason for different pattern — never uncaught exceptions.
- `import * as g from '@/guards/built-in'

## Documentation

- All exported and internal functions, classes, and types must have JSDoc docstrings. Keep them concise. 80% coverage enforced by CodeRabbit.
- When adding or modifying a library module or built-in spark, update both inline JSDoc and the corresponding file in `docs/`.

## Testing

Use Bun's test runner. Coverage threshold: 100%. Test helpers are in `@/core/lib/test-helpers` — see `.claude/testing-reference.md` for APIs and patterns. Tests should be DRY — extract shared setup and assertions into file-local helpers.
